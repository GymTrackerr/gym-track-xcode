Global constraints (always include)
-----------------------------------
You are working inside an existing SwiftUI + SwiftData project called GymTracker.

You MUST follow BOTH spec docs strictly:
1) notes_import_architecture.md
2) notes_import_implementation.md

The architecture doc defines constraints and must NOT be violated.
The implementation doc defines structure and details.

Hard constraints:
- All changes must be non-destructive.
- Do NOT change existing required fields’ types.
- Do NOT make SessionRep.weight/count optional.
- Cardio is ExerciseType.run | .bike | .swim via ExerciseType.isCardio (no .cardio case).
- SessionRep.weight remains TOTAL weight used for workload (rep.weight * rep.count stays).
- Existing routines must NOT have template exercises/order/state mutated by import.
- If a routine is newly created during the same import commit, template population is allowed for that newly created routine only.
- All resolver/duplicate/writer logic MUST be scoped to currentUserId; no global queries.

Behavior constraints:
- Parsing/preview is allowed without userId, but commit is blocked until currentUserId exists.
- Batch import: per draft all-or-nothing. One draft failure does not block other drafts.
- Writer transaction: create objects → try context.save(); on failure context.rollback(); keep draft editable with suggestions.
- Time parsing: strict formats only:
  - 24-hour `HH:mm-HH:mm` (two-digit hour/minute both sides)
  - 12-hour with AM/PM on both sides (case-insensitive, spaces optional)
  - reject mixed/ambiguous formats
- Device timezone; if end < start then +1 day.
- No silent commit-time date/time defaults; unresolved date/time must be explicitly resolved before commit.
- Weightless sets persistence: weight=0, unit=defaultWeightUnit (fallback .lb), add rep note.

UI constraints:
- Paste input uses SwiftUI TextEditor (multiline).
- Parse occurs only on “Parse” button tap (no live parsing on change).

Return format:
- Return only the changed/new files for the phase.
- Include a short “Build verification checklist” for what I should test after this phase.
- Ask me any MAJOR questions that block progress BEFORE writing code; otherwise make reasonable defaults consistent with the spec.

PHASE 3 — Parser skeleton + core parsing (PURE SWIFT)
================================================
Implement Phase 3 ONLY.

Rules:
- No SwiftData reads/writes.
- No UI changes.
- Pure Swift logic.
- Deterministic; no background tasks.

Tasks:
- Create NotesImportParser.swift with API:
  parseBatch(from: String, defaultWeightUnit: WeightUnit) -> NotesImportBatch
  parseSingleSession(from: String, defaultWeightUnit: WeightUnit) -> NotesImportDraft

Parsing requirements:
- Batch splitting by date header lines.
- Date formats: “December 15, 2025” and “Nov 15, 2022”; handle optional “, legs” routine suffix.
- Time range parsing: strict 24h `HH:mm-HH:mm` OR strict 12h with AM/PM on both sides (cross-midnight supported).
- Strength lines:
  - numbered prefix “1.” optional
  - parse multiple set segments: “2x10, 205lbs, 1x10, 225lbs”
  - parse “sets of”: “3 sets of 10”
  - parse rest: “1:30m rest” → restSeconds
  - per-side + bar: “35kg per side, 20kg bar” → store baseWeight/perSideWeight/isPerSide and ALSO compute total weight for persistence later.
- Cardio lines (run/bike/swim keywords + distance/duration):
  - parse “5km, 29min”; “1km, 5:15min”; “20min, 2km”; optional pace tokens like “9:36av”
  - store cardio fields on ParsedCardioSet (distance/duration/pace + distance unit)
- Unknown lines collected (do not discard).

Hash requirements:
- Generate importHash per session block AFTER splitting, using canonical normalization described in spec.

Output:
- NotesImportParser.swift (+ any small helper file NotesImportParsingHelpers.swift if needed).

Build verification:
- Add a small debug-only unit test or sample invocation in a separate DEBUG-only file (no UI integration yet),
  OR provide an example snippet I can run in a playground to validate parsing.

Major questions to ask me before coding:
- If any ambiguity about date parsing locale (assume English month names unless I say otherwise).
- Whether to treat “Started watch…” as note line (default yes; append to session notes).

- Must require userId.
- Must block commit if date/time are unresolved.
- Require explicit user resolution for missing date and missing time-range in UI before commit.

Tasks:
- Create NotesImportWriterService.swift with:
  commit(draft: NotesImportDraft, resolution: ..., userId: UUID, context: ModelContext, defaultWeightUnit: WeightUnit) throws -> Session
- Persistence rules:
  - Strength: create SessionEntry/SessionSet/SessionRep(s)

  - timestamp mapping: per spec; if end missing set timestampDone = timestamp + 60m with warning in notes.
  - Append leftover lines and warnings into Session notes (do not lose data).
- Duplicate detection:
  - Query sessions for userId + importHash match; return a duplicate flag for UI prompting (writer should not silently skip).
