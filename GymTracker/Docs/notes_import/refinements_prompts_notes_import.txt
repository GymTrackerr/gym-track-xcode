R1 — UX Defaults + Presentation (full-screen import, no auto-pick)
===========================================================
Goal:
- Import from Notes should open as a page/full-screen, not a small slide-over.
- Routine resolution must NOT auto-select the first existing routine when no match exists.
- Exercise resolution must NOT auto-select an existing exercise when no match exists.

Tasks:
1) Present import UI as `fullScreenCover` (preferred) or a navigation push, not a small sheet.
2) Routine resolution default:
   - If matched routine exists: show Matched
   - Else default mode should be Create (prefilled with parsed name) OR None
   - Must never silently pick first existing routine
3) Exercise resolution default:
   - If matched exercise exists: show Matched
   - Else default mode should be Create (prefilled with raw name)
   - Must never silently pick an existing exercise

Acceptance:
- Opening Import from Notes shows a full-screen experience.
- Unmatched routine/exercise never auto-picks an existing item.

R3 — Parser Hardening (headers, bar default, sequential reps)
===========================================================
Goal:
Improve parsing for real-world note formats.

Tasks:
1) Header parsing:
   Support these header styles:
   - “Pull, February 20, 2025”
   - “September 30, 2025 - Back and bicep”
   Rule: detect date anywhere in header line; routine name is leftover text around it.
2) “bar” token:
   - If “bar” appears without explicit weight:
     - default bar weight depends on defaultWeightUnit:
       - kg => 20
       - lb => 45
   - Store baseWeight and compute total rep.weight accordingly.
   - Still keep rep.weight as TOTAL used for workload.
3) Sequential reps / fractional reps:
   Example: “1x6.5”
   - Since SessionRep.count is Int, floor non-integer reps and add a warning/rep note.
4) Chained segments:
   Example:
   “Chest press, 1x5, 130 pounds, 1x6.5, 2x5, 110pounds (1:30m rest)”
   - Implement robust token grouping so NxR pairs get the nearest following weight token.
   - Rest applies to all sets in that line unless a different rest appears later.
5) Update/extend the existing DEBUG harness (or add one) with at least 5 regression samples from your notes.
   Print parsed output and ensure no crash.

Acceptance:
- The provided header examples parse date + routine correctly.
- “bar” results in baseWeight set and total weight correct.
- Chained segments produce the correct number of sets and weights.
- Fractional reps do not crash and produce a warning.

R3.1 — Cardio Telemetry Parsing (Cycling / Running Extras)
===========================================================
Goal:
Improve parser support for structured cardio telemetry while keeping the data model minimal and stable.

Tasks:
1) Core cardio parsing (existing behavior must remain):
   - Detect duration tokens: “X:YYm”, “Xmin”, “X:YY” -> durationSeconds.
   - Detect distance tokens: “N(.N)?km” or “mi” -> distance + distanceUnit.
   - Detect speed tokens: “N(.N)?km/h” or “mph”.
     - If avgSpeed field exists -> populate it.
     - Else -> append speed info to set notes.
2) Telemetry token detection (store in notes):
   - “###W” -> append “Power: ###W”
   - “##RPM” -> append “Cadence: ## RPM”
   - “level ##” -> append “Level: ##”
   - Any structured but unrecognized token -> append raw to notes.
3) Notes handling:
   - Preserve original order of telemetry data.
   - Never discard telemetry information.
   - Do NOT prompt user to choose important metrics (v1 behavior).
4) Example to support:
   “1. Indoor cycle, 181W, 71RPM, 34.1KM/H, level 50, 4:01m, 2.25km”
   Expected:
     - durationSeconds = 241
     - distance = 2.25 (km)
     - notes contain Power, Cadence, Speed, Level info.
5) Extend DEBUG/parser harness with at least one regression test for this exact example.

Acceptance:
- Duration and distance parsed correctly.
- Telemetry preserved in notes (or avgSpeed if implemented).
- No crash or dropped tokens.
- Existing cardio parsing behavior remains unchanged.

R3.2 — Header Normalization (Comma-Heavy & Flexible Date Formats)
===========================================================
Goal:
Support real-world headers with extra commas and varied date positioning.

Tasks:
1) Pre-normalize header line for date extraction:
   - Replace commas with spaces.
   - Collapse multiple spaces.
   - Trim leading/trailing whitespace.
2) Date detection rules:
   Support formats:
   - “January 22, 2025”
   - “January 22 2025”
   - “January, 22, 2025”
   - “Pull, February 20, 2025”
   - “September 30, 2025 - Back and bicep”
   - “Push, January, 22, 2025”
   Date may appear anywhere in header line.
3) Routine extraction:
   - After extracting date substring, remove it from header.
   - Trim punctuation and whitespace.
   - Remaining text becomes routineNameRaw.
   - If routineNameRaw is empty and next line is short text (e.g., “Push day”),
     use it as fallback routine name.
4) Time range behavior remains unchanged:
   - Support “HH:mm-HH:mm”.
   - Cross-midnight handling as already defined.
   - Device timezone.
5) Add regression samples:
   - “Push, January, 22, 2025”
   - “Pull, February 20, 2025”
   - “September 30, 2025 - Back and bicep”

Acceptance:
- Date parses correctly in all listed formats.
- Routine name extracted correctly.
- No “Date could not be parsed” warning for supported formats.
- Existing header formats remain unaffected.

## R6 — Post-R5 Fixes (Cardio set edit UI, live refresh, routine delete safety)
===========================================================

### A) Cardio set edit UI cleanup

* Current inline editing UI for cardio sets is messy.
* Improve layout to be clean and consistent with strength set UI:

  * Group fields (duration, distance+unit, pace) in a compact form.
  * Use appropriate input controls (e.g., numeric fields, pickers).
  * Ensure it is readable in both normal + edit modes.
* Do not change data model.

### B) Live refresh after import (no app restart)

Problem: after importing, new exercises/routines don’t show until app restart.

* Fix so exercises and routines lists refresh immediately after import commits.
* Use SwiftData best practices:

  * rely on @Query where possible
  * or publish change notifications / refresh triggers in services/viewmodels
* Do not introduce polling.
* Ensure Sessions list + Exercise picker + Routine picker reflect new items immediately.

### C) Routine deletion must not delete unrelated sessions

Problem: deleting a routine sometimes deletes sessions (even unrelated).

* Find and fix the relationship/delete rules causing cascade deletion.
* Requirement:

  * Deleting a Routine must NOT delete any Session.
  * Sessions that referenced the deleted routine should become routine=nil (or equivalent safe state).
* Do not break existing session deletion behavior.
* Add a minimal regression check:

  * Create two routines and two sessions (only one linked to routine A).
  * Delete routine A.
  * Verify both sessions still exist; only routine link is removed for the one that referenced it.

Output:

* Only changed files.
* Build verification checklist for A/B/C.
